# 代码审查 · 中级(P2) 与 低级(P3) 问题

**生成日期**：2026-06-09
**审查范围**：约 1.2 万行 Python（PyQt6 GUI + ADB/Scrcpy 设备控制 + OpenCV 模板匹配）
**来源**：两份独立审查（Claude 静态深读 + GPT 含实测运行）合并去重；所有条目均经源码核验。

> 本文件只列中级与低级问题。阻断级(P0) 与 高级(P1) 问题见 `代码审查_P0阻断与P1高级问题.md`。
> 这些问题不阻断核心功能，建议按排期处理。

## P2 · 中级（Medium）— 可维护性、健壮性与潜在风险

### M1. `HybridDeviceAdapter` 的 `get_resolution` / `click` 是潜在地雷（死方法）
- **位置**：`device_adapter.py:300-301`（`get_resolution`）、`:319-320`（`click`）
- **证据**：两个方法都委托到底层不存在的实现——`AdbAdapter` 无 `get_resolution`（`adb_utils` 有模块级函数但从未被 `AdbAdapter` 封装），全体 adapter 无 `click`（只有 `tap`）。grep 全项目确认：这两个方法当前**没有任何调用点**，故不是现行崩溃，但一旦被调用即 `AttributeError`。
- **影响**：潜在地雷，未来误用即崩。
- **修复**：给 `AdbAdapter` 补 `get_resolution()`（封装 `adb_utils.get_resolution`）；把 `click()` 改为 `tap()` 的别名或删除；基类补默认实现。
- **备注**：与 P1-B3 的 `swipe_path` 同源（接口不一致），但 `swipe_path` 有真实触发路径故列为阻断级，这两个无调用故列中级。

### M2. `internalize_all_images` 不处理 `multi_match` 模板
- **位置**：`script_model.py:198`
- **证据**：内化逻辑只遍历 `find_and_tap` / `wait_image` 的 `template` 字段（`:198`），不处理 `multi_match.templates` 列表与嵌套 `sub_actions` 中的外部路径。最小驱动验证：multi_match 的外部绝对路径不会被内化为项目内相对路径。
- **影响**：导入/迁移含多图匹配的循环脚本时，图片引用可能失效（指向开发机绝对路径）。
- **修复**：扩展内化逻辑到 `multi_match.templates` 及其 `sub_actions`（递归处理嵌套动作的模板字段）。

### M3. ADB 后台任务对象随会话累积不释放
- **位置**：`gui/main_window.py:534 / :555 / :607`
- **证据**：`_restart_adb` / `_refresh_devices` / `_detect_resolution` 都把 `AdbTask` 实例赋给同一属性 `self._adb_task`，链式触发时后者覆盖前者。
- **澄清**：`AdbTask(parent=self)` 由 Qt 父对象持有，**不会发生「GC 回收运行中线程」崩溃**（注释「保持引用防止 GC」表述误导）；真实问题是任务对象随会话累积、不显式释放。
- **修复**：`task.finished.connect(task.deleteLater)`，或用列表持有并在各自 finished/error 槽里移除自身。

### M4. `import config` 触发全盘扫描副作用
- **位置**：`config.py:410`
- **证据**：模块级 `_scanned_adb = scan_adb_paths()`，任何 `import config`（`image_engine` 等）都会在导入时遍历注册表 + WMIC 查询 + 全盘符 `glob`，耗时且有副作用、难测试。
- **修复**：改为惰性初始化（首次访问 `ADB_PATH` 时扫描）或显式 `init()` 调用。

### M5. JSON 反序列化无类型校验
- **位置**：`script_model.py` 的 `from_dict` 多处（`:27-35`、`:66-76`、`:135-145`）
- **证据**：全部用 `.get` 带默认值但不校验类型。`params` 被存成 list、某 `actions` 元素非 dict、`max_loops` 是字符串等，都会延迟到执行期才以 `AttributeError` / `TypeError` 爆出（如 `run_loop` 的 `int > str`）。
- **修复**：加载时做基本类型断言或 schema 校验，非法数据在加载阶段拒绝并给出明确错误。

### M6. 路径遍历：外部 JSON 的脚本名/模板名直接拼接路径
- **位置**：`script_engine.py:364`、`script_model.py:267`、`script_engine.py:161-163`
- **证据**：`script_project` / `template` / `target_name` 来自脚本 JSON，未规范化校验。`script_project="../../something"` 可跳出 `Scripts/` 加载任意目录的 `script.json`。
- **影响**：本地工具威胁有限，仍应防御。
- **修复**：`os.path.normpath` 后校验结果仍位于 `SCRIPTS_ROOT` / `pictures_dir` 之内，拒绝含 `..` 的名称。

### M7. `closeEvent` 未清理群控通道与 WorkflowRunner
- **位置**：`gui/main_window.py:1105-1123`
- **证据**：关闭窗口只停了 `_workers` 与 `_screencap_worker`，未处理 `self._group_adapters`（仍持 scrcpy 连接/子进程）与运行中的 `self._workflow_runner`。
- **影响**：退出后残留 scrcpy server 进程/socket，WorkflowRunner 线程可能在解释器退出时报 "Destroyed while running"。
- **修复**：`closeEvent` 遍历 `_group_adapters` 调 `client.stop()` 清空；若 `_workflow_runner` 在运行则 `stop()` + `wait()`。

### M8. 多处重操作在 GUI 主线程同步执行
- **位置**：`gui/device_selector.py:338`（设备扫描）、`gui/template_gallery_dialog.py:427`、`gui/tabs/library_tab.py:412`（找图测试）、`gui/convert_dialog.py:432-582`（转换向导）
- **证据**：`EmulatorManager.scan_all()`、`image_engine.match_all()`、多次 `cv2.matchTemplate` 均在主线程同步跑（其中 device_selector 的 docstring 还写「后台刷新」与实现矛盾）。
- **影响**：点击对应操作时界面冻结。
- **修复**：迁移到 `QThread`，结果经 `pyqtSignal` 回主线程更新。可参照 `gui/workers.py` 正确范例。

### M9. 大量重复 UI 构建代码已引发行为漂移
- **位置**：
  - `FlowLayout` / `_ImageCard` / `_display_name` 两套几乎重复：`gui/template_gallery_dialog.py:34-98` vs `gui/tabs/library_tab.py:30-106`
  - `_calc_layout` / `_widget_to_original` 重复：`gui/widgets.py:84-128,284-298` vs `gui/multi_region_widget.py:125-163`
  - 三个 Tab 的列表/属性逻辑高度雷同
  - QImage→cv 转换三处重复
  - `find_and_tap` 与 `wait_image` 属性面板大段复制：`action_props.py:154-220` vs `:223-271`
- **已造成的实害**：图库「取消」语义三处不一致——`workflow_tab.py:676-690` 把空列表当清空，与 `script_tab` / `loop_script_tab` 行为不符，用户点「返回」未选模板时会清空已配置模板。
- **修复**：抽公共基类/工具模块（`_flow_layout.py`、`_qimage_cv.py`、Tab mixin），统一「取消」语义。

### M10. QImage `bits()` 缓冲被 numpy 引用，潜在悬空内存
- **位置**：`gui/template_gallery_dialog.py:406-411`、`gui/tabs/library_tab.py:391-396`、`gui/convert_dialog.py:39-43`
- **证据**：`ptr = q_img.bits(); arr = np.array(ptr).reshape(...)` 依赖源 QImage 在转换完成前存活；某些 PyQt6/numpy 组合下 `np.array(sip.voidptr)` 可能创建视图而非拷贝。
- **影响**：低概率读到已释放内存（花屏/崩溃）。
- **修复**：显式拷贝并确保源 QImage 生命周期。参照 `gui/workers.py:78-80,115-117` 已用 `.copy()` 的正确做法。

### M11. 单文件超 800 行 / 函数过长
- **位置**：`gui/main_window.py`(1123行)、`gui/tabs/script_tab.py`(967行)；`MainWindow._init_ui`(~260行)、`_apply_styles`(~110行)、`convert_dialog._on_finish`(~150行)、`script_engine._execute_action`(~300行) 等。
- **影响**：可维护性差，正是 B1/H1 这类 bug 滋生的土壤。
- **修复**：按职责拆分方法；样式抽到独立常量；`_execute_action` 用 dispatch 表按 `action_type` 分发到子方法。

## P3 · 低级（Low）— 一致性、清理与风格

### L1. 用户可见品牌仍是旧通用名
- **位置**：`gui/main_window.py:81`、`:95`、`gui/top_bar.py:111`
- **证据**：窗口标题/顶栏仍显示「模拟器 · 自动化脚本」，而 README / 打包产物已是「镜界自动化」。
- **修复**：统一窗口标题、顶栏、关于页品牌为「镜界自动化」。

### L2. `loop_start` / `loop_end` 半实现
- **位置**：`script_engine.py:281`（仅警告跳过）、`gui/action_props.py` 注册表与 `format_action_text` 有文案，但各 Tab 过滤掉、无构建器。
- **证据**：注册了、能格式化、但无法添加也无属性面板；引擎只警告跳过。循环体被当普通步骤只执行一次，无报错（静默功能缺陷）。
- **修复**：要么实现区间解析，要么从 `ACTION_REGISTRY` 与 `format_action_text` 一并移除，避免误导。

### L3. scrcpy 解码线程是 daemon，stop 不保存引用 / 不 join
- **位置**：`scrcpy_client.py:308`（启动 daemon thread）、`:313`（stop 关 socket/进程但不 join）
- **证据**：解码线程读 socket 与 `stop()` 关 socket 存在竞态；`_video_socket` 置 None 后解码线程若走到 `setblocking` 会 `AttributeError`（不被 `except OSError` 捕获，逃逸出线程）。
- **修复**：保存线程引用并在 stop 中短超时 join；`_stream_loop` 内对 socket 取本地引用再用，或放宽 `except (OSError, AttributeError)`。

### L4. 工作流多设备执行共享同一批 ActionNode 对象（潜在）
- **位置**：`workflow_runner.py:150-151`
- **证据**：创建临时模型时浅拷贝 steps。当前执行器主要读取参数，故非已证实 bug。
- **修复**：若后续执行器会写 `ActionNode.params`，需为每台设备深拷贝步骤。

### L5. 空 `except: pass` 吞异常
- **位置**：`gui/main_window.py:985`、`script_model.py:184`、`bot_worker.py:131`、`scrcpy_client.py:321-323/331-332/345-346/652-654`、`emulator_manager.py:430-431`
- **影响**：回调吞异常会让监听器里的真实 bug 永久消失；`wait_adb_ready` 把所有错误当「未就绪」掩盖配置错误。
- **修复**：至少 `logger.debug(exc_info=True)` 记录；`except:` 改 `except Exception:`（避免吞 `KeyboardInterrupt`）。

### L6. 死代码 `_add_test_find_btn`
- **位置**：`gui/action_props.py:708-781`
- **证据**：全代码库无任何调用点（grep 确认）。导致「找图点击」动作无测试按钮，与 tap/swipe（有测试按钮）不一致；函数内 `_log` 定义后未用，实际走 `print()`。
- **修复**：要么在 `build_find_and_tap_props` 末尾调用并改 `print`→`_log`，要么删除约 70 行死代码。

### L7. 雷电模拟器端口纯公式推导，无交叉验证
- **位置**：`emulator_manager.py:124-125`（雷电 `5555 + index*2`）
- **证据**：MuMu 已用 `adb devices` 实际端口交叉验证（`:193`），但雷电完全没有，`running=True` 但端口错时会拼出连不上的 `device_id`，特定多开布局下可能控错实例。
- **修复**：雷电也用 `adb devices` 实际端口交叉验证，而非纯公式。

### L8. `assert` 做运行时状态校验
- **位置**：`scrcpy_client.py:291`（`assert not self.alive`）
- **证据**：`python -O` 下 assert 被剥离，状态校验失效。
- **修复**：改 `if self.alive: raise RuntimeError(...)`。

### L9. `screencap_to_memory` 的换行替换可能损坏二进制 PNG
- **位置**：`adb_utils.py:118`
- **证据**：`raw_data.replace(b'\r\n', b'\n')` 是 Windows adb shell 换行污染的 workaround，但会把 PNG 数据中合法的 `0x0D0A` 字节序列也误改，导致解码失败。这正是 `screencap_fast`（pull 二进制）存在的原因。
- **修复**：`screencap_to_memory` 仅作最后兜底（当前 `AdbAdapter.get_frame` 已 fast 优先、to_memory 兜底，设计正确），文档标注其不可靠性。

### L10. `set_display_power(on)` 实为 toggle，语义与签名不符
- **位置**：`device_adapter.py:151-154`
- **证据**：`AdbAdapter.set_display_power(on)` 忽略 `on` 参数，无条件发 `KEYCODE_POWER`（翻转）。调用方误以为幂等会出错。
- **修复**：改名为 `toggle_display_power`，或在 docstring 强调限制。

### L11. 魔法数字与函数内重复 import
- **位置**：遍布——`script_engine.py:200/465`（循环内 `import cv2`/`traceback`）、各 Tab 反复 import `QListWidgetItem`/`time`、`main_window.py` 大量 `setFixedSize(50,50)`/各类 `wait(5000)` 超时、`widgets.py` 拖拽阈值 `>10`、端口公式常量等。
- **修复**：import 提到模块顶部；魔法数字抽到 `constants.py` 命名常量（`SYNC_INTERVAL_MS`、`WORKER_STOP_TIMEOUT_MS`、`SWIPE_THRESHOLD_PX`、`MAX_COORD` 等）。

### L12. `convert_dialog` 用无参 `disconnect()` 清空按钮全部连接
- **位置**：`gui/convert_dialog.py:373`、`:428`
- **证据**：`self.btn_next.clicked.disconnect()` 断开所有连接再重连，将来加别的连接会被一并清掉。
- **修复**：保存具体连接对象用 `disconnect(slot)` 精确断开，或单槽 + 状态变量分派。

### L13. `script_tab` 通过 `hasattr` + 私有属性深度耦合主窗口
- **位置**：`gui/tabs/script_tab.py:280-304, 360-375, 826-831`
- **证据**：直接访问 `main_win.live_sync_btn`、`main_win._on_toggle_live_sync`、`main_win.screenshot_widget._recording_mode` 等，靠 `hasattr` 防御；主窗口字段改名即静默失效。
- **修复**：在 `MainWindow` 暴露语义化公共方法（`enter_recording_mode()` 等），子 Tab 通过信号或这些接口交互。
