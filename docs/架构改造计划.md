# 下一轮架构改造计划

> 生成日期：2026-06-11
> 背景：第三轮代码审查（全项目五路并行审查）后，机械性修复已全部完成并提交
> （4 个用户反馈 Bug + 全部 P1 + 机械性 P2/P3 + 4 项安全加固，共约 35 项，涉及 24 个文件）。
> 本文档收录**有意推迟**的架构性改造项——这些问题修复成本高、需要先做设计决策，
> 不适合与机械性修复混在同一批变更中。

## 推迟原则

以下各项满足至少一条：

- 需要拆分文件或调整模块边界（影响面大，需回归测试）
- 需要先做设计讨论（存在多种可行方案，选错返工成本高）
- 改动会改变现有行为语义（不是"修错"而是"换做法"）

## 二、script_engine 模板路径解析统一（P2 · 工作量：小～中）

**现状**：`find_and_tap`（~162-172 行）与 `wait_image`（~244-259 行）各自实现
三级路径回退（项目 Pictures → 绝对路径 → templates/），与已有的
`_resolve_template_path`（~427 行）逻辑不一致。当 template 是纯文件名且三处都
找不到时静默跳过，没有诊断信息。

**方案**：三处统一收敛到 `_resolve_template_path`，找不到文件时输出全部候选
路径的诊断日志。

**风险**：路径回退顺序是隐式契约，统一前需先写覆盖三种来源（项目内/绝对/相对）
的回归测试，确认现有脚本不受影响。

## 三、找图诊断的重复 matchTemplate 优化（P3 · 工作量：小）

**现状**：`find_and_tap` 匹配循环内（~202-207 行），`image_engine.match_all`
未命中时又执行一次裸 `cv2.matchTemplate` 仅为了拿最高分做诊断输出。超时轮询
期间（默认 5s / 0.3s 间隔）每帧的匹配开销翻倍。

**方案**：给 `image_engine.match_all`（或新增 `match_best_score`）增加返回
最高分数的能力，诊断直接复用，不再重复卷积。

**依赖**：需要改 `image_engine` 的公共返回值结构，所有调用方同步适配。

---

## 四、workflow_runner 改用 engine.run()（P2 · 工作量：中）

**现状**：`_run_on_device` 手动迭代 `script_model.actions` 调用私有方法
`engine._execute_action`，绕过了 `ScriptEngine.run()` 的完整生命周期
（统一异常处理、失败计数、收尾逻辑）。本轮已临时补上 `_pre_flight_checks()`
显式调用，但仍依赖两个私有方法。

**方案**：让 `ScriptEngine.run()` 支持外部传入停止/暂停控制（目前已支持
`interrupt_check` / `pause_event`），`_run_on_device` 改为直接调 `run()`，
删除手动迭代。

**风险**：`run()` 内部的失败重试/弹窗守卫行为会首次作用于工作流场景，
需要确认多设备并发下日志与回调的归属正确。

---

## 五、scrcpy adb reverse 就绪探测（P2 · 工作量：小）

**现状**：`_deploy_server` 用 `time.sleep(0.5)` 硬等 `adb reverse` 生效
（scrcpy_client.py ~434 行），ADB 负载高时不够，连接失败无自愈。

**方案**：启动 server 前用 `adb reverse --list` 轮询验证隧道存在
（最多重试 3 次，每次 300ms），失败给出明确错误信息。

---

## 六、滑动录制分辨率检查异步化（P2 · 工作量：小～中）

**现状**：`on_recorded_swipe` → `_check_recording_resolution` 在主线程同步调用
ADB `get_resolution`（script_tab.py ~394-400 行），每次滑动卡顿数百毫秒。
点击路径已有 `_RecordClickWorker` 异步化，滑动路径没有。

**方案二选一**：
1. 录制开始时检查一次并缓存结果，整个录制会话内不再查询（推荐，最简单）；
2. 复用主窗口 `_detect_resolution` 已缓存的分辨率标签值。

**备注**：若做第一节的录制控制器提取，此项应合并进去一起做。

---

## 七、_save_region 的 exec() 重入防护（P1 遗留 · 工作量：中）

**现状**：`gui/widgets.py` 的 `_save_region` 用 `while not saved:` 循环内嵌
`dlg.exec()`（嵌套事件循环）。scrcpy 实时帧更新等外部事件可能在对话框期间
重入鼠标事件，污染 `_drag_start/_drag_end` 状态。

**方案**：改为非阻塞对话框 + `finished` 信号回调（Qt 推荐做法），或最低限度
在进入 `_save_region` 时屏蔽鼠标追踪、finally 中恢复并复位拖拽状态。

**风险**：改信号回调会重构保存重试流程（当前靠 while 循环实现"名称为空重试"），
需要重新设计交互。

---

## 八、Temp 快照清理机制（新功能 · 需先讨论设计）

**现状**：录制时每次点击的快照 `step_<毫秒时间戳>.png` 永久留在项目 Temp
目录，没有任何清理机制，长期使用无限膨胀。

**待讨论的设计决策**：

1. **清理时机**：停止录制时？保存脚本时？启动应用时？
2. **保留判定**：快照被 tap 节点 `snapshot` 参数引用即保留——但"升级为找图
   点击"后原 tap 节点可能被替换，引用关系如何追踪？
3. **是否提供 UI**：给用户一个"清理未引用快照"按钮，还是全自动？

**建议**：保存脚本成功后，扫描 Temp 目录中未被当前模型任何节点引用的
`step_*.png` 并删除（带日志）。先实现最保守版本，观察使用情况再决定是否自动化。

## 九、私有属性访问遗留清理（P3 · 工作量：小）

本轮已为 `MultiRegionWidget` / `ScreenshotWidget` 增加 `get_source_image()` 等
公共访问器并迁移了 4 个调用点。剩余的跨组件隐式协议：

- `script_tab._upgrade_to_find_and_tap` 直接设置 `sw._click_marker`（widgets.py
  的 paintEvent 隐式读取）
- `main_window.closeEvent` 直接复位 `screenshot_widget._live_control_mode` /
  `_scrcpy_adapter`
- 录制模式开关直接写 `screenshot_widget._recording_mode`

**方案**：为 `ScreenshotWidget` 定义正式的模式切换 API
（`set_click_marker()` / `set_recording_mode()` / `detach_live_control()`）。
