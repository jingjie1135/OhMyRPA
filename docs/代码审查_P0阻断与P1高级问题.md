## P0 · 阻断级（Blocker）— 必须修复，否则核心功能崩溃/失效

### B1. `wait_image` 动作必崩（NameError）
- **位置**：`script_engine.py:260`
- **证据**：
  ```python
  img = screencap_to_memory(self.device_id)   # 该名字从未 import
  ```
  文件顶部（`script_engine.py:1-8`）只导入了 `image_engine / template_meta / device_adapter`，`wait_image` 块内也无局部 import。实测触发 `NameError`，被外层 `except` 吞掉打印 traceback，脚本中止。
- **影响**：任何含「等待图片」动作的脚本一进此分支即崩溃，功能完全不可用。
- **修复**：改为 `img = self._adapter.get_frame()`，与同文件 `find_and_tap`（`script_engine.py:185`）保持 `DeviceAdapter` 抽象一致。

### B2. `run_script` 几乎必然变无限循环
- **位置**：`script_engine.py:391`
- **证据**：
  ```python
  if getattr(cfg, 'max_loops', 0) > 0 or getattr(cfg, 'scan_interval', 0) > 0:
      sub_engine.run_loop()
  ```
  `ScriptConfig.scan_interval` 默认 `1.0`（`script_model.py:44`、`:74`），所以 `scan_interval > 0` 对任何子脚本恒成立；而默认 `max_loops=0` 在 `run_loop` 里表示无限循环（`script_engine.py:444`）。
- **影响**：被引用的子脚本一律被当成循环脚本永久执行，**父脚本永远卡在第一个 `run_script`，再也不往下走**。`run_script` 在绝大多数配置下不可用。
- **修复**：循环判定不能依赖默认非零的 `scan_interval`。引入显式 `loop_mode` 标志，或仅当 `max_loops > 0` / 用户明确开启循环时才 `run_loop()`。

### B3. `HybridDeviceAdapter.swipe_path` 参数不匹配，实时控制路径必 TypeError
- **位置**：`device_adapter.py:325-326`
- **证据**：
  ```python
  def swipe_path(self, path, duration_ms=500):
      self._actuator.swipe_path(path, duration_ms)   # 传 2 个参数
  ```
  但 `AdbAdapter.swipe_path(self, path)`（`device_adapter.py:130`）与 `ScrcpyAdapter.swipe_path(self, path)`（`device_adapter.py:212`）都只收 1 个参数 → `TypeError`。
- **触发点**：`script_engine.py:117-119`，当 `supports_touch=True`（scrcpy 实时/录制路径激活）执行高精度轨迹滑动时必崩。
- **影响**：GUI 实时控制路径下的轨迹滑动不可用。（注：`workflow_runner` 走纯 ADB，`supports_touch=False`，落到普通 `swipe` 分支，不触发此 bug。）
- **修复**：统一 `DeviceAdapter / AdbAdapter / ScrcpyAdapter / HybridDeviceAdapter` 四处 `swipe_path()` 签名（底层加 `duration_ms` 参数，或 Hybrid 委托时不透传）。

### B4. 循环模式「启用模板」配置完全不生效（用户禁用的规则仍被执行）
- **位置**：`script_engine.py:419` / `:451`
- **证据**：
  - `loop_script_tab.py:534` 返回用户勾选的启用模板；
  - `bot_worker.py:79-80` 把它传给 `engine.run_loop(self.enabled_templates)`；
  - 但 `script_engine.py:419` 的 `def run_loop(self, enabled_templates=None)` **收下参数后从不使用**，`:451` 仍 `for action in self.model.actions` 执行全部动作。
- **影响**：用户在循环脚本里取消勾选某模板，以为已禁用，实际仍会点击。属安全/正确性问题（自动化规则失控）。
- **修复**：在 `run_loop()` 中按 `enabled_templates` 过滤未启用动作；若不打算支持，则移除 UI 勾选能力，避免误导。

### B5. 默认 unittest discovery 在无设备环境直接失败
- **位置**：`test_scrcpy.py:9-20`
- **证据**：模块级代码在 import 阶段就执行 ADB 扫描，无设备时 `sys.exit(1)`：
  ```python
  if not device_id:
      print("No device")
      sys.exit(1)
  ```
  实测 `python -m unittest discover -p "test*.py"` 失败：`SystemExit: 1 / No device`。
- **影响**：默认测试套件（含 `test_project_cleanup.py` 等正常用例）在任何无设备环境下整体跑不起来，CI / 本地快速测试不可用。
- **修复**：把 ADB/scrcpy 逻辑移入测试函数体内；无设备时用 `unittest.skip`；或将文件改名为非默认发现的手动集成测试（如 `manual_scrcpy_check.py`）。

## P1 · 高级（High）— 正确性、并发与资源问题，应在合并前修复

### H1. 多尺度回退分支 score 错位 → IndexError 或读到错误分数
- **位置**：`image_engine.py:220 / :226-227 / :234`
- **证据**：微尺度循环把匹配结果存进 `res`（`:220`），命中后只更新了 `locations` 和 `tmpl_w/h`（`:226-227`），**没有更新 `result`**。随后：
  ```python
  score = float(result[pt_y, pt_x])   # result 仍是旧的 1.0 尺度结果
  ```
  - 缩放 0.98 / 0.99（模板变小 → `res` 形状比 `result` 大）→ `locations` 索引越界 → **IndexError**；
  - 缩放 1.01 / 1.02 → 读到错误位置的置信度。
- **影响**：跨模拟器存在微小缩放偏差时（正是该回退分支存在的理由）匹配反而异常或给出错误置信度。
- **修复**：命中后同步 `result = res`。

### H2. `clear_cache` 重复定义，生效版本无锁，与并发加载竞态
- **位置**：`image_engine.py:23`（带锁）与 `:82`（无锁，覆盖前者）
- **证据**：同名函数定义两次，后者覆盖前者。生效的第二版直接 `pop/clear` `_template_cache` 而不持 `_cache_lock`，而 `load_templates`（`:45-54`、`:76-77`）在持锁下并发读写同一字典。
- **影响**：`workflow_runner` 多设备并行 + `internalize_image` 触发清缓存（`script_model.py:183`）时存在数据竞态，偶发崩溃或缓存错乱。
- **修复**：删除 `:82` 的重复定义，保留 `:23` 的带锁版本。

### H3. 群控启动阻塞 GUI，并从非 GUI 线程更新 UI / 共享状态
- **位置**：`gui/main_window.py:942 / :962 / :964 / :970 / :975`
- **证据**：`_on_toggle_group_control` 在按钮 handler 中：
  - `:942` 设等待光标（仅掩盖卡顿）；
  - `:970` 启动裸 `threading.Thread`；
  - `:975` 立即 `for t in threads: t.join()` 在主线程同步等所有 scrcpy 连接建立 → **UI 冻结**；
  - worker 内 `:962` 并发写 `self._group_adapters` dict、`:963` `nonlocal success_count` 非原子自增、`:964/966` 从非 GUI 线程直接调 `self._append_log()` 操作 `QTextEdit`（Qt 未定义行为）。
- **影响**：界面冻结 + 跨线程 GUI 访问偶发崩溃 + 计数/字典竞态。
- **修复**：整体移入 `QThread`/worker，日志与 `_group_adapters` 更新通过 `pyqtSignal` 回主线程；不要在 GUI handler 中 `join`。

### H4. scrcpy `start()` 失败时设备端 server 进程 / socket 泄漏
- **位置**：`scrcpy_client.py:289-311 / :438-444 / :467-507`
- **证据**：`start()` 顺序为 `_deploy_server()` → `_init_connection()` → `alive=True`。若 `_init_connection()` 在 accept 阶段超时抛 `ConnectionError`，此时 `alive` 仍为 False，调用方通常只在 `alive` 时才 `stop()`，导致已 `Popen` 的设备端 server 进程不被 terminate；`_video_socket`（若第一个 accept 成功、第二个超时）也不关闭（`finally` 只关 `_listen_socket`，`:506`）。
- **影响**：每次连接失败残留一个设备端 `app_process` server 进程；反复重连堆积僵尸 server，占用设备资源并可能导致后续 reverse 冲突。
- **修复**：`start()` 用 `try/except` 包裹，失败路径调用 `self.stop()` 或显式清理 `_server_process` / `_video_socket`。

### H5. 嵌套 `run_script` 无循环引用 / 递归深度防护
- **位置**：`script_engine.py:356-400`
- **证据**：`run_script` 加载子脚本并递归调用 `_execute_action`，没有任何「已访问脚本集合」或递归深度上限。A→B 且 B→A（或自引用）将无限递归直到 `RecursionError` / 栈溢出。与 B2（子脚本进无限循环）叠加更不可控。
- **影响**：用户误配循环引用即崩溃。
- **修复**：在引擎中维护 `call_stack` / `visited` 集合，执行前检测 `script_project` 是否已在调用链中，命中则报错跳过；并设最大嵌套深度。

### H6. `loop_script_tab` 重复连接 `itemChanged` 信号
- **位置**：`gui/tabs/loop_script_tab.py:328`
- **证据**：`self.action_list.itemChanged.connect(self._on_item_check_changed)` 写在每次都会执行的 `_reload_action_list_ui` 内部，每切换/导入一次脚本项目就新增一条连接，从不断开。
- **影响**：第 N 次加载后，勾选一个模板会触发回调 N 次；随使用时间增长性能下降。典型信号泄漏。
- **修复**：把这条 `connect` 移到 `_init_ui` 里只连一次。

### H7. 打包脚本混入运行时数据边界
- **位置**：`build.bat:19 / :31`，对照 `.gitignore:36/37/41/43`
- **证据**：`.gitignore` 标记 `targets/ popups/ Scripts/ Workflows/` 为运行时数据；但 `build.bat:19` 的 robocopy `/XD` 只排除 `Daily Scripts`，**未排除 `Workflows / targets / popups`**；`build.bat:31` 还显式 `--include-data-dir=targets=targets`，把用户运行时模板打进单文件 exe。
- **影响**：打包产物混入开发机上的用户运行时数据，体积膨胀且可能泄漏私有脚本/模板；运行时数据与内置资源边界混乱。
- **修复**：明确区分「项目内置资源」与「用户运行时数据」。若无内置模板，robocopy 排除 `targets/ popups/ Workflows/` 并移除 `--include-data-dir=targets=targets`。
